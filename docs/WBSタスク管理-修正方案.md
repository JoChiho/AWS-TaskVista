# TaskVista：基于 WBS 视角的任务管理修正方案

| 项目 | 内容 |
|------|------|
| 版本 | 0.1（草案·实现前的方针整理） |
| 日期 | 2026-07-21 |
| 对象 | 项目内的任务（Work Package）管理 |
| 前提 | 不破坏现行 DynamoDB / API / 画面，分阶段引入 |
| 状态 | 待确认（由方案确认人审定即可） |

---

## 1. 背景与目的

### 1.1 现状问题（对 PM 意见的理解）

现行 TaskVista 的任务被设计为 **项目下的扁平作业卡片**。

- 没有父子关系与分解结构（无法表达 WBS 的「交付物 → 工作包 → 作业」）
- 画面主轴是 **状态看板**（未着手 / 进行中 / レビュー待ち / 完了 / 保留）
- 虽有工数、开始日、截止日期、进度，但无法从结构上看出 **「整体分解了多少、彼此依赖什么」**
- 要么把大需求塞进一张任务，要么并列同名任务，**范围边界容易模糊**

PM 提出的「希望以 **WBS（Work Breakdown Structure）** 的观点来设计 task 管理」，可理解为：

> 把项目分解为交付物与作业单位，并在这些单位上管理责任、工数、进度与依赖。  
> 状态管理是 WBS 的 **运营层**，而不是结构本身。

### 1.2 本方案目标

1. 将 **WBS 树**（项目内的层级结构）作为一等概念引入  
2. **不破坏** 现行扁平 Task，分阶段扩展为「带层级的 Task」  
3. 看板 / 表格 / 时间线 / レビュー 与 WBS 协调共存  
4. 不加重小团队（数人～十几人）的运营负担（不以完整 PMBOK 为目标）

### 1.3 非目标（本版不做）

- 正式 EVM（CV / SV / CPI）或完整 baselining  
- 成本科目、会计联动  
- 跨项目的公共 WBS 模板共享平台  
- 初版就实现自动排程（关键路径优化引擎）

## 2. 现行数据结构整理

### 2.1 实体关系（现状）

```
Project (1) ──owns──> Task (N) ──has──> Comment / Attachment
                │
                ├── assignees[] / assigneeId
                ├── reviewers[]
                ├── status / priority / completionPercent
                ├── estimatedEffortDays / startDate / dueDate
                └── requirement / description
```

| 实体 | 主键 | 作用 |
|------|------|------|
| Project | `projectId` | 团队作业单位与成员边界 |
| Task | `projectId` + `taskId` | 作业卡片（扁平） |
| Comment | 挂在 task 下 | 讨论 |
| Attachment | task 元数据 + S3 | 资料 |

### 2.2 Task 属性（现状与 WBS 的对应）

| 现行字段 | 含义 | 在 WBS 中的位置 |
|----------|------|-----------------|
| `title` | 名称 | WBS 要素名 |
| `requirement` | 要件 / 验收意图 | 范围描述（接近交付物定义） |
| `description` | 补充说明 | 作业说明 |
| `status` | 工作流状态 | **运营状态**（不是结构） |
| `completionPercent` | 0–100% | 进度（叶子录入，父级可汇总） |
| `estimatedEffortDays` | 预计工数（人日） | 工数估算（原则在叶子） |
| `startDate` / `dueDate` | 日程 | 进度计划（引入依赖后与约束并用） |
| `assignees` | 担当 | 责任（近似简化版 RACI 的 R/A） |
| `reviewers` | レビュアー | 质量门禁担当 |
| `priority` | 优先级 | 执行优先（与 WBS 编号无关） |
| — | **父子 / 层级** | **未实现（本方案核心）** |
| — | **依赖关系** | **未实现** |
| — | **WBS 编码** | **未实现** |
| — | **要素类型** | **未实现**（交付物 / 工作包等） |

### 2.3 存储约束（实现前提）

Tasks 表（DynamoDB）：

| 类型 | 键 | 用途 |
|------|-----|------|
| PK / SK | `projectId` / `taskId` | 项目内列表 |
| GSI TaskIdIndex | `taskId` | 单条获取 |
| GSI AssigneeIndex | `assigneeId` + `dueDate` | 担当任务 |
| GSI ProjectStatusIndex | `projectId` + `status` | （参考） |

- 列表以 **按项目 Query** 为主 → 层级在 **应用侧组装** 最自然  
- 小规模项目：拉取全量任务后在客户端/服务端建树即可  
- 父子关系用 **Task 上的自引用字段** 即可（不必新建表）

## 3. WBS 设计方针（面向 TaskVista 的简化版）

### 3.1 采用的 WBS 思路

不做严格 PMBOK 式 WBS，而采用 **现场可用的 2～3 层交付物分解**。

```
Project（项目）
 └─ Level 1: 交付物 / 阶段（Deliverable or Phase）
      └─ Level 2: 工作包（Work Package）← 工数、担当、状态的主单位
           └─ Level 3: 子任务（可选·明细作业）
```

| 层级 | 名称（UI 方案） | 表示什么 | 工数 / 状态 |
|------|-----------------|----------|-------------|
| L0 | 项目 | 现有 Project | 仅汇总 |
| L1 | 大项 / 交付物 | 交付成果、大块工作 | 由子项汇总（原则不手输） |
| L2 | 工作包 | 担当实际执行的单位 | **主要在此管理** |
| L3 | 子任务 | 再分解的细节（可选） | 可为叶子时录入 |

**原则（100% 规则的简化版）**

- 父级范围应由子级合计覆盖（避免遗漏与重复）  
- **工数、进度的「录入」仅限叶子节点（无子节点）**  
- 父级工数、进度由 **子级自动汇总**（原则上不可覆盖）

### 3.2 「任务」一词的再定义

| 术语 | 含义 | 与现行 UI 的关系 |
|------|------|------------------|
| WBS 要素 / 节点 | 层级上的一个节点 | 数据上仍是同一条 `Task` 记录 |
| 工作包 | 执行与看板的主单位 | 约等于现行「任务」卡片 |
| 父级（汇总） | 用于结构与汇总 | 看板上折叠或可选不显示 |
| 叶子 | 没有子节点的节点 | 可编辑工数、开始日、状态 |

实现上采用 **一张表、一个实体（Task）增加层级字段**，避免实体膨胀。

## 4. 数据模型修正案

### 4.1 建议为 Task 增加的字段

```ts
interface Task {
  // ---- 现有（保留）----
  taskId: string
  projectId: string
  title: string
  description?: string
  requirement?: string
  status: TaskStatus
  priority: TaskPriority
  assignees?: TaskAssignee[]
  reviewers?: TaskReviewer[]
  completionPercent?: number
  estimatedEffortDays?: number
  startDate?: string
  dueDate?: string
  // ... attachments, createdBy, timestamps, isDeleted

  // ---- WBS 新增（本方案）----
  /** 父任务 ID。未设置 = 项目下的根节点 */
  parentTaskId?: string | null

  /**
   * 展示与排序用的 WBS 编码（如 "1", "1.2", "1.2.3"）
   * 可手改；未设置时也可由 sortOrder + 层级自动编号
   */
  wbsCode?: string

  /** 同一父级下的排序（从 0 起的整数） */
  sortOrder?: number

  /**
   * 节点类型
   * - summary: 仅汇总（通常有子节点；看板上可选显示）
   * - work_package: 执行单位（默认）
   * - milestone: 里程碑（工数 0、日期点）
   */
  nodeType?: 'summary' | 'work_package' | 'milestone'

  /**
   * 前置任务 ID 列表（可选·Phase 3）
   * 例：本任务开始前应完成的 taskId
   */
  predecessorIds?: string[]

  /**
   * 汇总缓存（可选·读优化）
   * 可在更新时重算，也可在展示时现算
   */
  rollup?: {
    childCount: number
    estimatedEffortDaysSum?: number
    completionPercentAvg?: number // 或工数加权平均
    earliestStartDate?: string
    latestDueDate?: string
  }
}
```

### 4.2 字段设计要点

| 项目 | 方针 |
|------|------|
| `parentTaskId` | 自引用。删除时对子节点 **上移** 或 **级联逻辑删除**（需定策略） |
| `wbsCode` | 给人看的编号。变更时可提供子孙重编号选项 |
| `sortOrder` | 拖拽排序的实体字段；可与 `wbsCode` 独立 |
| `nodeType` | 分流 UI 与汇总规则；省略时为 `work_package` |
| `predecessorIds` | Phase 3；初版可为空 |
| 深度上限 | **建议最多 3 层**（L1–L3）；超过则警告 |

### 4.3 与现有数据的兼容

| 现有 Task | 迁移后处理 |
|-----------|------------|
| 全部 | `parentTaskId = null`（根节点） |
| 全部 | `nodeType = 'work_package'` |
| 全部 | `sortOrder` 可按 `createdAt` 等编号 |
| 全部 | `wbsCode` 可后续批量生成（可选） |

**无破坏性变更。** 新属性未设置时，行为与现行扁平列表一致。

### 4.4 汇总规则（父节点）

以叶子的值为准，父级按如下 **推导**：

| 指标 | 汇总方法（推荐） |
|------|------------------|
| 预计工数 | 子级 `estimatedEffortDays` 合计（未设置视为 0） |
| 进度 % | **工数加权平均**；子级无工数时回退为等权平均 |
| 开始日 | 子级最小 `startDate`（参考展示） |
| 截止日期 | 子级最大 `dueDate`（参考展示） |
| 状态 | 示例：全部完了→完了 / 任一进行中→进行中 / 全未着手→未着手 / 混杂→进行中。**子级含レビュー待ち・保留时，父级视为进行中** |

> 父级的 `status` / `completionPercent` / `estimatedEffortDays` 是写入 DB 冗余，还是 API 响应时计算，属实现选择。  
> **推荐：** 只允许写叶子；父级展示值在 **读取时 rollup**（必要时用 `rollup` 缓存）。

### 4.5 对 DynamoDB 的影响

| 变更 | 是否需要 | 说明 |
|------|----------|------|
| 新表 | 不需要 | 扩展 Task 属性即可 |
| 新 GSI（初版） | 可选 | 若常按父级查子级可再议；小规模全量建树即可 |
| 现有 GSI | 保持 | AssigneeIndex 等不变 |
| 迁移 | 软迁移 | 新字段 optional；可用批处理补 `sortOrder` / `wbsCode` |

**若将来频繁按父级列子级：**

- GSI 方案：`ParentIndex` … PK=`projectId`，SK=`parentTaskId#sortOrder#taskId`  
- 或改造表 SK 为含 `parentTaskId` 的复合键（**改表主键成本高，不推荐**）

## 5. API 修正案

### 5.1 现有 API 扩展（向后兼容）

| 方法 | 路径 | 变更 |
|------|------|------|
| POST | `/projects/{id}/tasks` | body 可增加 `parentTaskId`, `nodeType`, `sortOrder` |
| PUT | `/tasks/{id}` | 同上 + `wbsCode`, `predecessorIds` |
| GET | `/projects/{id}/tasks` | 响应带层级字段；query 支持 `view=flat\|tree` |
| GET | `/tasks/{id}` | 可附带子项摘要（`children[]`，可选） |
| PATCH | `/tasks/{id}/status` | **仅允许叶子**（父级返回 400 或忽略） |

### 5.2 建议新增 API

| 方法 | 路径 | 用途 |
|------|------|------|
| POST | `/tasks/{id}/children` | 创建子任务（父 ID 固定在 URL） |
| POST | `/tasks/{id}/move` | 改父级 / 排序 `{ newParentId, sortOrder }` |
| POST | `/projects/{id}/tasks/reorder` | 同一父级下批量排序 |
| POST | `/projects/{id}/wbs/renumber` | WBS 编码重编号 |

### 5.3 校验（重要）

1. `parentTaskId` 必须存在于同一 `projectId` 内  
2. 禁止循环引用（不能把子孙设为自己的父级）  
3. 深度 ≤ 3（可配置）  
4. 里程碑不能有子节点（或允许但不推荐）  
5. 将父级标为「完了」仅当全部子级完了时（可选）  
6. 逻辑删除：父级删除时 **默认级联逻辑删除子级**（或拒绝删除并要求先迁移子级）

## 6. UI / 画面修正方针

### 6.1 各视图职责（引入 WBS 后）

| 画面 | 职责 | 与 WBS 的关系 |
|------|------|----------------|
| **WBS / 树**（新建·推荐） | 创建结构、分解、编号、排序 | **主编辑画面** |
| 表格 | 跨字段编辑与筛选 | 层级缩进，或筛选项「仅叶子」 |
| 看板 | 日常推进状态 | **默认仅叶子（工作包）** |
| 时间线 | 日程总览 | 叶子 + 可选父级汇总条 |
| 仪表盘 | 我的担当与レビュー | 以叶子为主（维持现状） |
| 任务详情 | 单节点全部属性 | 面包屑（父路径）+ 子列表 |

### 6.2 新画面方案：WBS 视图

```
[项目名]  WBS          [＋添加根节点]  [重编号]

1  ■ 要件定义（交付物）                    工数 5.0  进度 40%
1.1  □ ヒアリング                          2.0  100%  担当:A
1.2  □ 要件书作成                          3.0   0%   担当:B
2  ■ 设计
2.1  □ 画面设计 …
```

操作：

- 行的 **缩进 / 反缩进**（改父子）  
- **拖拽排序**（同一父级内）  
- 「添加子项」「同级添加」  
- 点击打开现有详情抽屉  

### 6.3 与看板的共存

问题：若把父级（汇总）放进看板，列含义会混乱。

**推荐规则**

| 设置 | 内容 |
|------|------|
| 默认 | 看板仅显示 `nodeType != summary` 且为 **叶子** 的节点 |
| 选项 | 「也显示父级」开关（进阶用户） |
| 卡片展示 | 可选小字显示 WBS 编号（如 `1.2`） |
| レビュー待ち | 仍按叶子单位（与现行一致） |

### 6.4 与时间线的共存

- 叶子：与现行相同，`startDate` + `estimatedEffortDays`  
- 父级：用子级 min(start)～max(end) 画浅色条（Phase 1.5）  
- 拖拽设置开始日：**仅叶子**（保持现行交互）

### 6.5 用语（产品 UI 为日语时的对照）

| 概念 | UI 显示方案（日语产品） | 中文理解 |
|------|-------------------------|----------|
| WBS 视图 | **WBS** 或 **構成** | 结构 / WBS |
| 根 / L1 | **大項目** | 大项 |
| L2 | **作業**（可与现行「タスク」等同） | 工作 / 任务 |
| L3 | **サブタスク** | 子任务 |
| wbsCode | **WBS 番号** | WBS 编号 |
| nodeType=milestone | **マイルストーン** | 里程碑 |
| rollup 进度 | **進捗（集計）** | 进度（汇总） |

无需全面改名「タスク」。在说明中写清「任务 ≈ 工作包」即可。

## 7. 分阶段导入路线图

### Phase 0：确认与定稿（当前）

- 审定本方案（层级深度、父级汇总规则、看板范围）  
- 定义成功指标（例：单项目中使用 2 层及以上分解的比例）

### Phase 1：数据 + 最小 UI（MVP）

**后端**

- Task 增加 `parentTaskId`, `sortOrder`, `wbsCode`, `nodeType`  
- 扩展 create / update / list  
- 循环与深度校验  
- 现有任务仍作为根节点工作  

**前端**

- 表格增加缩进列或 WBS 编号列  
- 创建任务时可「选择父级」  
- 详情中展示面包屑与子列表  
- 看板仅过滤叶子  

**验收条件**

- 未使用 WBS 的既有项目体验不被破坏  
- 可创建、列表、详情查看 2 层父子  
- 父级进度可由子级汇总展示  

### Phase 2：WBS 视图 + 操作

- 专用 WBS 树画面（加入导航）  
- 缩进 / 排序 / 重编号  
- 时间线父级条显示  

### Phase 3：依赖与里程碑

- `predecessorIds` 与简易依赖展示  
- 里程碑（日期点）  
- （可选）破坏依赖时的开始日警告  

### Phase 4：模板与高级汇总（可选）

- 创建项目时的 WBS 模板  
- 基于工数的仪表盘（按 WBS 剩余工数）  
- 导出（CSV / 大纲）

| 现行功能 | 引入 WBS 后 |
|----------|-------------|
| 看板拖拽改 status | 仅叶子；规则本身不变 |
| 进度滑块 | 仅叶子；父级只读汇总 |

| 逻辑删除 | 在 API 中明确父子策略 |

## 9. 备选方案对比

| 方案 | 内容 | 优点 | 缺点 | 判定 |
|------|------|------|------|------|
| **A. Task 自引用（本方案）** | 在 Task 上增加 `parentTaskId` 等 | 迁移易、API 简单、复用现有能力 | 极深树或超大项目时客户端建树压力 | **采用** |
| B. 独立实体 WbsNode | 结构与执行任务分离 | 概念更干净 | 双重维护、界面复杂、迁移成本高 | 初期不采用 |
| C. 标签模拟层级 | 如 `epic` 标签 | 实现最轻 | 不是真 WBS，难满足 PM 要求 | 不采用 |
| D. 项目多层化 | 子项目 | 权限边界清晰 | 对小团队过重、跨列表困难 | 不采用 |

## 10. 风险与对策

| 风险 | 影响 | 对策 |
|------|------|------|
| 层级拆得过深，运营变重 | 使用率下降 | 深度上限 3；默认仍可扁平 |
| 父级进看板导致混乱 | 状态语义不清 | 默认仅叶子 |
| 汇总与手输不一致 | 信任下降 | 父级 UI 禁止手输 |
| 删除 / 移动产生孤儿节点 | 数据不一致 | move API 与删除策略必须有测试 |
| GSI 投影再次漏字段 | 新属性读不到 | 列表继续走基表 Query（现行方针） |
| 中英日术语不一致 | 培训成本 | 用语表（§6.5）同步到功能说明 |

## 11. 成功指标（草案）

| 指标 | 目标意象 |
|------|----------|
| 既有项目回归 | 不使用 WBS 时传统 UX 不坏 |
| 结构使用率 | 上线 1 个月后，新项目 ≥50% 使用 2 层及以上 |
| 范围讨论质量 | 「超大单任务」平均体量下降（可主观评估标题粒度） |
| 进度可解释性 | 例会可直接使用父级汇总进度 |

## 12. 建议确认的决策点

实现前建议至少确认以下事项（可由方案确认人直接定稿）：

1. **层级深度**：2 层是否够用，是否需要到 3 层  
2. **看板范围**：是否默认仅叶子  
3. **父级进度**：是否用工数加权平均（还是简单平均）  
4. **删除父级时**：级联删除子级，还是禁止删除  
5. **初版范围**：仅 Phase 1，还是连 Phase 2（WBS 专用画面）一起做  
6. **里程碑与依赖**：是否进初版（本方案建议放在 Phase 3）  

---

## 13. 总结

- 现行作为 **扁平执行卡片** 完成度较高，但缺少 **分解结构（WBS）**。  
- 修正核心是 **Task 自引用（`parentTaskId`）+ 排序 + 类型 + 汇总规则**，无需拆表。  
- 画面上采用 **WBS 树建结构，看板 / 时间线跑叶子** 的双层用法，对小团队最合适。  
- 建议在 **保持向后兼容** 的前提下分阶段推进（Phase 1 → 2 → 3）。
