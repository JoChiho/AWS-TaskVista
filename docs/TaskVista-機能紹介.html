1本システムの目的

TaskVista は、小規模なチーム向けのタスクおよびプロジェクト進捗の管理を支援する Web アプリケーションです。

チームで共有する「電子的なホワイトボード」や「待合せ帳」「簡易な日程表」に相当するものとお考えください。 同一のプロジェクト内で作業を分担し、担当を決め、進捗を更新し、コメントを残すことができます。 管理職や同僚も、プロジェクトの現在地を一覧で把握しやすくなります。

解決を目指す課題
チャットの履歴から作業内容を探し回る、誰が何をしているか見えにくい、 どこで止まっているかが分からない、残作業の目安が立ちにくい——といった状況を、 一つの画面に集約して整理することを目的としています。

2想定する利用者

  • 数名から十数名程度の小規模チーム・プロジェクト単位の組織
  • 複数のプロジェクトを並行して追う必要のある方
  • 自ら作業を進めつつ、他者の成果物の確認・レビューも行う方

プログラミングやクラウド技術に関する知識は不要です。 ブラウザでログインできれば、基本的な操作は可能です。

3一日の利用の流れ（例）

  1. ホームを開く——「自分の担当タスク」に期限が近いものがないか確認し、 あわせて「レビュー待ちのタスク」に自分宛てがないかを確認します。
  2. 対象のプロジェクトを開く——かんばんで状態を更新したり、 タイムラインで今週の日程を照合したりします。
  3. タスクの詳細を開く——進捗を調整し、説明を追記し、コメントや添付を残します。
  4. 他者の確認が必要な場合——状態を「レビュー待ち」にし、レビュアーを指定します。 レビュアーのホーム画面に当該タスクが現れます。

4プロジェクトの管理

まずプロジェクトを用意し、その中にタスクを作成します。 プロジェクトは、チームが共同で作業する単位です。

  • 作成——名称・説明・状態（例：計画中、進行中、完了）を設定します。
  • メンバーの追加——メールアドレス等により同僚を招待します（多くの場合、オーナーが実施します）。
  • 一覧での把握——ホームでは、各プロジェクトのタスク数や最近の更新状況を確認できます。
補足： プロジェクトの削除は、原則として画面上から見えなくする「論理削除」です。 直ちにデータを完全に消去する方式ではないため、運用上の安全性に配慮しています。

5タスクと進捗

タスクは、日々実際に取り組む「一件の作業」です。

タスクに含まれる主な情報

項目意味
タスク名作業内容が一目で分かる名称
要件 / 説明依頼内容・背景・注意点など
ステータス現在どの段階にあるか（下表参照）
優先度緊急度・重要度の目安
担当者実施する人（複数名の指定可）
レビュアー確認・レビューを行う人（主に「レビュー待ち」で使用）
進捗0％〜100％（進捗率）
開始日作業を開始する予定日（タイムラインの基準）
予定工数（人日）おおよそ何日分の作業量か
締切日提出・完了の期限
コメント / 添付議論の記録、資料ファイルなど

タスクのステータス

未着手 進行中 レビュー待ち 完了 保留

進捗とステータスの関係

  • 進捗を 0％ にすると、「未着手」に合わせることが多いです。
  • 進捗が 1％〜99％ の場合は、「進行中」に合わせることが多いです。
  • 進捗を 100％ にすると、「完了」に合わせることが多いです。
  • ただし レビュー待ち および 保留 は、 進捗の変更によって自動的に書き換えられません。 「確認中で最終確定していない」「いったん止める」といった運用に適しています。

6タスクの見え方（三つの画面）

同一のプロジェクトでも、目的に応じて表示方法を切り替えられます。

① かんばん

壁に貼った付箋のように、列ごとに状態を並べます。 カードを別の列へドラッグすると、状態が更新されます。 短い打合せや、素早い進捗更新に適しています。

② テーブル

一覧表形式で、並べ替え・絞り込み・検索ができます。 「誰のタスクか」「いつ締切か」「誰がレビュアーか」を調べる場合に適しています。

③ タイムライン

カレンダー上に、各タスクの期間を帯で示します。 いつからいつまで作業する予定かを俯瞰したい場合に適しています。

タスク詳細

任意のタスクを開くと、内容の編集、進捗の調整、 コメントや添付の確認など、最も詳しい情報を扱えます。

7タイムラインによる日程の把握

タイムラインは、「いつ着手し、およそ何日分の作業か」を視覚的に示す画面です。

  • 開始日と予定工数（人日）を設定すると、 カレンダー上に色付きの帯として表示されます。
  • 帯の長さはおおむね所要日数を、帯の内側の塗りは進捗を表します。
  • 締切日は期限の目印として表示できますが、 締切日から開始日を自動で逆算して帯を描くことはありません。

開始日が未設定の場合

画面上部に「日程未設定」の一覧が表示されます（原則として「保留」のタスクは含めません）。 該当タスクをカレンダー上の日付へドラッグすると、その日が開始日として保存されます。 ドラッグ中は他のタスクを一時的に隠し、日付だけを見やすくします。

要点： タイムラインは開始日を基準とします。 締切日のみを設定し、開始日がない場合、カレンダー上に工期の帯は表示されません。

また、「プロジェクト全体」と「自分が担当するタスクのみ」の表示切替にも対応しており、 情報量を抑えて確認できます。

8レビュアーとレビュー

作業の完了には、作成者本人の申告だけでなく、他者による確認が必要な場合があります。

  1. 実施者がタスクの状態をレビュー待ちに変更します。
  2. プロジェクトのメンバーから、レビュアーを一人または複数名指定します。
  3. レビュアーは、ホームの「レビュー待ちのタスク」や、かんばんの「要レビュー」から対象を確認します。
  4. 内容を確認し、問題がなければ完了へ進める（または差し戻す）運用が可能です。

状態が「完了」となった後も、レビュアーが設定済みであればその情報は保持して表示されます。 レビュアーが一度も設定されていない場合は「なし」と表示されます。

ホーム画面

「レビュー待ちのタスク」として、自分がレビュアーに指定されている案件のみが並びます。 選択すると、該当プロジェクトのかんばんを開き、そのタスクをすぐに確認できます。

かんばん画面

「要レビュー」を有効にすると、自分がレビューするカードだけに絞り込み、 見た目でも強調表示されます。見落としを防ぎやすくなります。

9ダッシュボード（ホーム）

ログイン後に最初に表示される画面で、一日の作業の起点として利用できます。

区分表示内容
プロジェクト概要 参加中のプロジェクト、状態、タスク数、最近の更新の有無など
自分の担当タスク 未完了で、自分が担当（または共同担当）となっているタスク。締切の順に整理しやすくなっています。
レビュー待ちのタスク 状態が「レビュー待ち」であり、かつ自分がレビュアーであるタスク。 担当タスクの欄の下に配置されています。

10メンバー・表示名・権限

  • ログイン——組織から案内された方法で、ブラウザ上からサインインします。
  • 表示名——姓・名を登録できます。 一覧やアバターでは、分かりにくい識別子ではなく氏名が表示されるよう努めています。 アバターには、多くの場合「姓」のみを表示します。
  • プロジェクトのメンバー——原則として、参加しているプロジェクトの内容のみを閲覧・操作できます。
  • オーナー——メンバーの招待など、管理に関する操作をより広く行える立場です。
安全に関する考え方： リンクを知っているだけでは全データにアクセスできません。ログインが必要であり、 かつ自分が参加しているプロジェクトに限定されます。 添付ファイルも、無期限に誰でも取得できる形では公開しません。

11コメントと添付ファイル

コメント

タスクごとにやり取りを残せます。 チャットの履歴に埋もれにくく、後から経緯を追いやすくなります。

添付ファイル

画面キャプチャや文書など、タスクに関連するファイルを登録できます。 タスクと一体で管理されるため、後続の担当者も資料を見つけやすくなります。

12利用上の注意

  • 組織から案内された正式な接続先（本番の画面など）をご利用ください。
  • ログインの有効期限が切れた場合は、再度サインインが必要です。これは通常の安全対策です。
  • タスクの更新やかんぼんの操作は、おおむね直ちに保存されます。 通信状態が悪い場合は、しばらく待つか画面を再読み込みしてから再度お試しください。
  • ホームの「レビュー待ち」などが表示されない場合は、システムの更新に伴う設定の反映漏れの可能性があります。 操作ミスとは限りませんので、管理担当者へご連絡ください。
まとめ
TaskVista は、小規模チームにおいて 「何をするか・誰がするか・どこまで進んだか・いつ行うか・誰が確認するか」 を一箇所に集約するためのツールです。 かんばんで進め、テーブルで探し、タイムラインで日程を見渡し、ホームで今日取り組むべき事項を把握する—— という使い方を想定しています。

13今後の改善・開発予定

現時点の機能に加え、以下の方向で順次改善・拡張を検討しています。 内容・優先順位は利用状況に応じて変更する場合があります。

No.予定する内容狙い
1 レビュアーへのリマインド通知 レビュー待ちのまま放置されにくいよう、レビュアーに「確認をお願いします」と知らせる仕組みを想定しています。
2 プロジェクトメンバーへの役割・権限の割り当て 誰が何を操作できるかを役割ごとに分け、誤操作を減らしつつ、責任の所在を明確にすることを目指します。

3 工数とメンバーの予定を踏まえたタスクの自動割り当て 各タスクの所要見積りと、メンバーの空き状況を照合し、 「誰に振るか」の候補を自動で示すことを検討しています。
