1システム概要

Findix Chat は、ブラウザ上で動作する AI チャット／画像生成／社内文書検索のための Web アプリケーションです。 本書は AWS 版（GCP 版 findix-chat-web をフォークし、認証・ホスティング・バックエンドを AWS へ全面移植したもの）を対象とします。 Firebase 依存は一切含まず、GCP 版とは別リポジトリで並行運用しています。

フロントエンド findix-chat-web-aws

Vue 3 (TSX) + TypeScript + Vuetify 3 の SPA。Amplify Hosting または S3 + CloudFront で配信。 認証は Cognito Hosted UI（OAuth フェデレーション）。

バックエンド findix-chat-api-aws

API Gateway (HTTP API) + Lambda (Python) + Amazon Bedrock。Terraform で IaC 管理。 ai-tool 単一環境のデモレベル最小構成（navino は対象外）。

主な機能

機能説明エンドポイントBedrock モデル
チャットAI との対話（直近6件の履歴を送信）POST /v1/ai-chatClaude (Haiku 4.5 / Sonnet 4.6)

社内検索社内文書 PDF への RAG 検索POST /v1/internal-searchClaude + Titan Embeddings (KB)

2全体構成図

ブラウザ（SPA）から AWS バックエンドまでの全体像です。認証は Cognito、API は JWT 検証付きの API Gateway を経由して Lambda → Bedrock へ到達します。

①ブラウザがホスティングから SPA を取得 → ②③ Cognito Hosted UI でログインし ID トークンを取得 → ④ API 呼び出し時に Authorization: Bearer <IDトークン> を付与 → ⑤ API Gateway の JWT オーソライザが Cognito の署名を検証 → ⑥ 該当 Lambda を invoke → ⑦ Lambda が Bedrock で推論／RAG 検索を実行して応答を返却。

5デプロイモード／操作モード

デプロイモード（プロダクト別ビルド）

用途別に2つのプロダクトをビルドし、それぞれ .env.<mode> で切り替えます。VITE_APP_ENV がランタイムの識別子です。

モードプロダクト起動コマンド

ai-tool社内 AI ツール（チャット + 画像生成 + 社内検索）npm run serve:ai-tool
バックエンド findix-chat-api-aws は ai-tool モードのみを対象としています（navino は対象外）。

操作モード（ai-tool 内、選択モデルで切替）

選択中の AI モデルに応じて操作モードと呼び出しエンドポイントが決まります（MODEL_CATEGORIES マッピング）。

6認証フロー

認証は Cognito User Pool + Hosted UI（OAuth フェデレーション）。Google / Microsoft の signInWithRedirect で Hosted UI へ遷移し、戻り時に ID トークンを受け取ります。

関数 / ファイル役割
utils/modules/amplify.tsAmplify.configure（VITE_APP_COGNITO_* から設定）。env 未設定時はスキップ
utils/modules/auth.tssignInWithGoogle / signInWithMicrosoft / logOut / getCurrentUserEmail / isApprovedDomain
permission.tsグローバル router ガード。ログイン判定とドメイン許可リスト検査
api/request.tsfetchAuthSession() の ID トークンを Authorization: Bearer で付与
重要: Hosted UI は誰でもサインインを完了できるため、フロントのメールドメイン判定は UX 用。 実際の制限は バックエンド側でも強制します（API Gateway の JWT 検証に加え、各 Lambda が email クレームの @findix.co.jp ドメインを再チェック）。

7リクエストフロー

チャット送信を例にした、フロント内部からバックエンドまでの流れです。

ポイント

  • 履歴: composable が直近 6 件の会話を送信。テキストは先頭+末尾で 1000 文字に切り詰め（getChatHistory）。
  • 送信形式: multipart/form-data（inputText / chatHistory / model ＋ 画像は imageSize / aspectRatio）。
  • 呼び分け: VITE_APP_AI_TOOL_API_BASE をベースに /v1/ai-chat・/v1/ai-image・/v1/internal-search を選択。
  • レスポンス契約: フロントが response.data.body.answer / error.response.data.body.detail を読むため、バックエンドは { "result": "...", "body": { ... } } でラップして返す。

8社内検索（RAG）構成

社内文書 PDF を Bedrock Knowledge Base（保存先は S3 Vectors）に取り込んで検索します。 取り込み（埋め込み生成・チャンク分割）は KB が担うため、Lambda は依存ゼロのまま retrieve_and_generate を呼ぶだけです。

取り込みは手動 sync: PDF を S3 にアップロード後、aws bedrock-agent start-ingestion-job を実行する必要があります （S3 イベント自動連携は未実装）。ステータスが COMPLETE になれば検索可能。文書を追加・差し替えたら都度 sync を再実行します。

主な Terraform リソース（knowledge_base.tf）

リソース用途
aws_s3_bucket.docs社内文書の原本（KB データソース）。パブリックアクセス全ブロック
aws_s3vectors_vector_bucket / aws_s3vectors_index埋め込みベクトルの保存先（cosine 距離、Titan の次元に一致）
aws_bedrockagent_knowledge_base取り込み・検索を担う KB 本体
aws_iam_role.kbBedrock サービスが assume するロール（埋め込み呼び出し・S3 読み取り・S3 Vectors 操作）

※ S3 Vectors を使う KB は AWS provider 6.27 以降が必要（本リポジトリは 6.50）。

9バックエンド（AWS / Terraform）

API Gateway（HTTP API）

  • JWT オーソライザ: Cognito User Pool のトークンを署名検証（issuer = Cognito、audience = App Client ID）。
  • ルート: POST /v1/ai-chat / POST /v1/ai-image / POST /v1/internal-search（すべて JWT 認可必須）。
  • CORS: allowed_origins に登録した Amplify URL のみ許可（メソッドは POST/OPTIONS）。
  • スロットリング: ステージ全体に rate/burst 制限（denial of wallet 対策）。

Lambda（3 関数・共通設定）

関数ハンドラ主な環境変数
{name}-ai-chatai_chat.handler.lambda_handlerBEDROCK_CHAT_MODEL_ID / BEDROCK_CHAT_MODELS / ALLOWED_EMAIL_DOMAIN
{name}-ai-imageai_image.handler.lambda_handlerBEDROCK_IMAGE_MODEL_ID / ALLOWED_EMAIL_DOMAIN
{name}-internal-searchinternal_search.handler.lambda_handlerBEDROCK_KB_ID / BEDROCK_RAG_MODEL_ARN / KB_NUMBER_OF_RESULTS / ALLOWED_EMAIL_DOMAIN

共通: runtime python3.12 / timeout 60s / memory 512MB / reserved_concurrent_executions による同時実行キャップ / CloudWatch Logs（保持日数は変数化）。

state 管理

Terraform state は S3 バケット findix-chat-terraform（ap-northeast-1）+ use_lockfile ネイティブロック（DynamoDB 不要）。 バージョニング有効化を推奨（誤操作からの復旧用）。

10環境変数（.env.<mode>）

UI カスタマイズに加え、AWS 連携に以下を使用します。AWS 構築前はプレースホルダ値が入っており、実値の記入が必要です。

変数用途
VITE_APP_ENVランタイム識別子（navino / ai-tool）
VITE_APP_COGNITO_USER_POOL_IDCognito User Pool ID
VITE_APP_COGNITO_USER_POOL_CLIENT_IDApp Client ID（public, no secret）
VITE_APP_COGNITO_DOMAINHosted UI ドメイン
VITE_APP_COGNITO_REDIRECT_SIGN_IN / ..._SIGN_OUTログイン/ログアウト後リダイレクト先（Cognito の Callback/Logout URL と一致）
VITE_APP_CHAT_API_BASE / VITE_APP_AI_TOOL_API_BASEバックエンドのベース URL（navino / ai-tool）
VITE_APP_AUTH_ENABLED認証の有効/無効
VITE_APP_AUTH_VALID_DOMAIN許可するメールドメイン（カンマ区切り）
VITE_APP_AUTH_PROVIDER表示するログインプロバイダ（Google / Microsoft）
VITE_APP_USABLE_FEATURES利用可能な操作モード
VITE_APP_WEB_SOCKET_URL音声入力（speech-to-text）用 WebSocket URL
VITE_APP_FEEDBACK_ENDPOINTフィードバック送信先
Cognito の env がプレースホルダ（空）の間は Amplify.configure がスキップされ、実際のログイン／チャットは動作しません。 UI だけ確認したい場合は一時的に VITE_APP_AUTH_ENABLED=false でトップ画面を表示できます。

11デプロイ手順

1. バックエンド（先）

cd envs/ai-tool
cp terraform.tfvars.example terraform.tfvars
terraform init
terraform apply

# API URL を取得
terraform output -raw api_base_url
# → https://xxxx.execute-api.ap-northeast-1.amazonaws.com

前提: AWS CLI 認証済み / Terraform ≥1.10 / state 用 S3 バケット作成済み / Bedrock のモデルアクセス有効化済み。

2. フロント（後）

# .env.ai-tool に API URL を記入
VITE_APP_AI_TOOL_API_BASE = 'https://xxxx...'

# ビルド + zip 化
npm run package:ai-tool
# → ai-tool-dist.zip を生成

配信は手段A（推奨）: Amplify Console「Deploy without Git provider」で zip アップロード。 手段B: aws s3 sync dist/ s3://<bucket> --delete + CloudFront invalidation。

SPA catch-all rewrite が必須: 200 (Rewrite) → /index.html。 無いと /login への直リンクやリロードで 404 になります。S3 + CloudFront では custom error response（403/404 → /index.html, 200）で代替します。
