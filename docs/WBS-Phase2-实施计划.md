## 1. 目标与非目标

### 1.1 目标

1. **日程视图产品化**：将现行「时间线/日历条」升级为 **WBS 感知的甘特图**（左树 + 右时间轴）。
2. **结构编辑主战场**：新增 **構成 (WBS)** 屏：缩进/反缩进、同级排序、加子/加同级、重编号。
3. **持久化同级顺序**：`sortOrder` 在構成（及必要时甘特树序）中可写回。
4. 不破坏 Phase 1：父只读汇总、深度 ≤3、看板默认叶子、3×2 详情。

### 2.1 用户可见变化

### 2.2 布局（自研）

```
┌──────────────┬────────────────────────────────────────────┐
│ 左：タスク列  │ 右：時間軸（日/週 スケール）                 │
│ WBS · 标题   │  [==== leaf bar ====]  ▶予定終了旗          │
│ ▸ 1 大项     │  [~~~~ parent rollup ~~~~]（只读）          │

```

- 复用并演进：`TaskTimeline.vue` + `taskSchedule.ts` + `TaskTimelineView.vue`。
- sticky：顶日期头、左任务列（现有能力加强）。
- 缩放：日宽（现有 `dayWidth`）+ 可选「今日」滚动定位。
- 范围：仍由任务 planned 范围推算；可加「前后 padding 天」。

### 2.3 日程数据模型与条编辑语义（重要）

**条的几何（与 P1 一致）**

- 起点 = `plannedStartDate`
- 长度（天）= `estimatedEffortDays`（缺省 1）
- `plannedDueDate` = **旗标**，不决定条长（除非后续产品改规则；P2 不改）

| 手势 | 写回字段 | 规则 |
|------|----------|------|
| **拖整条** | `plannedStartDate` | 工数不变 → 条整体平移；旗标不自动跟（除非另做开关，默认不跟） |
| **拖右缘** | `estimatedEffortDays` | 起点不变；长度 ≥ 0.5 或 1 的步长（建议 **0.5 人日** 或 **1 日**，实现时统一） |
| **拖左缘** | `plannedStartDate` + `estimatedEffortDays` | **推荐：保持条右端（结束日）不动** → 改开始且反算工数。若工数会 &lt; 最小则钳制 |
| **点击条/行** | — | 打开详情 |
| **未设定 → 日期格** | `plannedStartDate` | 保持现有 DnD |
| **父条** | 禁止写 | 任何拖动手势 no-op 或禁止抓取 |

父条几何：

- 开始 = rollup `plannedStartDate`（子 min）
- 结束优先 = rollup `plannedDueDate`（子 max）；若无则用子条推算 end 的 max
- 样式：低对比、虚线或半透明；无进度拖拽把手

### 2.4 树与过滤

| 能力 | 说明 |
|------|------|
| 默认顺序 | 与表 WBS 树一致：根 → 子 DFS，`sortOrder` 同级 |
| 展开状态 | 本地 `expandedIds`（可 `sessionStorage` 按 projectId） |
| 全体 / 自分 | 保持；自分时父若无匹配子可隐藏或显示「空壳父」（推荐：**有可见子孙才显示父**） |
| 担当/优先/状态过滤 | 保持；过滤后树修剪 |
| 仅叶子模式 | **可选** toggle（默认 **树含父**，因甘特要看汇总条） |

## 3. P2b：WBS 構成视图 + 结构 API

### 3.1 导航与路由

| 项 | 方案 |
|----|------|
| 路由 | `/projects/:projectId/wbs`（`name: 'task-wbs'`） |
| 文案 | **構成** 或 **WBS**（日语 UI 推荐 **構成**，副标题 WBS） |
| AppNav | 第 4 个视图按钮（建议图标 `mdi-file-tree`） |
| 入口 | 项目内切换；表格「在構成中打开」可选 |

### 3.2 画面能力（MVP）

**形态（产品确认）**：独立「構成」屏为 **左→右树状思维导图**（方案 A 大纲），不是第二张表。  
テーブル = 属性横断；構成 = 分解结构的可视化与结构编辑。

```
  [プロジェクト]
       ├── [1 要件定義] ── [1.1 ヒアリング]
       │                └─ [1.2 要件整理]
       └── [2 設計] …
```

| 操作 | 行为 |
|------|------|
| 展开/折叠 | 节点 chevron；全体 展開/折りたたみ |
| **同级拖拽排序** | ⋮⋮ 把手 + vuedraggable → `reorder` API |
| **缩进（字下げ）** | 变为直前同级的子（深度 ≤3）→ `move` |
| **反缩进（字上げ）** | 提升为父的同级 → `move` |
| **＋子 / ＋同阶** | 打开创建表单预填 `parentTaskId` |
| **重编号** | 按当前树 DFS 重写全部 `wbsCode` |
| 点击节点 | 详情 |
| 节点摘要 | WBS、标题、予/実工数、进度、status |

### 3.3 后端 API（P2b）— **已实现（开工前准备）**

| 方法 | 路径 | 用途 | 状态 |
|------|------|------|------|
| `POST` | `/tasks/{id}/move` | `{ newParentId, sortOrder? }` — 环/深度/renumber | ✅ |
| `POST` | `/projects/{id}/tasks/reorder` | `{ parentTaskId?, items: [{ taskId, sortOrder }] }` | ✅ |
| `POST` | `/projects/{id}/wbs/renumber` | 全树重编号 + sortOrder 0..n-1 | ✅ |
| `POST` | `/tasks/{id}/children` | 创建子任务（父固定） | ✅ |

说明：内部复用 `updateTask` / `createTask` 校验；前端 `api/tasks.ts` + `stores/tasks` 已封装。  
**生产反映：** `npm run deploy:backend`（Lambda + 不足ルート同期）。

### 3.4 与表格的关系

| 表（Table） | 構成（WBS） |

| 层内「＋子」可保留 | 構成为权威结构编辑入口 |

## 6. 风险与对策

| 风险 | 对策 |
|------|------|
| 条左缘与「工数定长」反直觉 | 固定采用「右端不动」；tooltip 说明 |
| 自研甘特拖拽在触控/缩放下抖 | 指针 capture + 网格吸附（1 日） |
| 大项目行数多（数百） | 虚拟滚动列为 P2 后期可选；先保证 &lt;200 行流畅 |
| 構成与表双入口状态不一致 | 统一走 tasks store + recomputeWbs |
| move/reorder 并发 | 按 project 串行或版本号；先简单 last-write-wins |
| 甘特与構成工作量膨胀 | **严格 P2a→P2b**；P2a 不阻塞於構成 API |

## 7. 成功标准（Phase 2 整体）

1. 用户能用 **ガント** 看清 WBS 时间结构，并直接改叶子开始/工数。  
2. 用户能在 **構成** 调整层级与顺序并重编号，无需只靠表格。  
3. 父汇总条只读；与 P1 规则无回归。  
4. 文档与方案状态更新为 Phase 2 已交付。
